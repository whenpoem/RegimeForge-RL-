# RegimeForge Architecture

RegimeForge is the public name of the project. The internal Python package remains `backend.regime_lens`.

## What the system is

RegimeForge is a synthetic market research platform for regime-aware reinforcement learning in trading.
It combines:

1. A configurable synthetic market generator with hidden regime dynamics.
2. Multiple trading agent variants, including a regime-aware mixture-of-experts model.
3. A training and evaluation pipeline that writes structured artifacts for analysis.
4. A terminal UI and experiment runner for inspection, comparison, and reporting.

## Synthetic market

The market is generated by `SyntheticMarketEnv` in `backend/regime_lens/market.py`.

At a high level, the environment works in two stages:

1. It samples a hidden regime sequence, normally using a Markov transition matrix.
2. It generates returns from regime-specific parameters such as drift, volatility, autocorrelation, and rare jumps.

The default regime set is:

- `bull`
- `bear`
- `chop`
- `shock`

Each regime has its own characteristics. For example, `bull` has positive drift, `bear` has negative drift, `chop` tends to be mean-reverting, and `shock` has higher volatility and jump risk.

The environment exposes:

- A compact observation vector with engineered features such as short-term return, medium-term return, volatility, trend gap, drawdown, current position, and unrealized PnL.
- An oracle observation mode that appends the true regime as one-hot input.
- A regime schedule override for curriculum and OOD-style experiments.

The result is a market process that is stochastic but still structured enough for regime-learning research.

## Agent variants

RegimeForge currently supports four agent types, defined in `backend/regime_lens/config.py`:

- `dqn`
  - Standard DQN baseline.
  - No explicit regime input, gate, or experts.

- `rcmoe_dqn`
  - Regime-aware mixture-of-experts DQN.
  - Uses a gating network to route observations to multiple experts.
  - This is the main research model.

- `oracle_dqn`
  - DQN with the true regime one-hot appended to the observation.
  - Serves as an upper-bound reference.

- `hmm_dqn`
  - Two-stage baseline.
  - Fits a regime detector first, then feeds the inferred state to a DQN policy.

The same action space is used across agents:

- `short`
- `flat`
- `long`

## Training pipeline

Training lives in `backend/regime_lens/training.py` and is orchestrated by `TrainingManager`.

The pipeline is:

1. Create a new run directory and persist the initial summary.
2. Configure runtime settings such as device, CPU threads, and process priority.
3. Instantiate the synthetic environment and the selected agent.
4. Roll out episodes, collect rewards, losses, and per-step trading statistics.
5. Flush metrics and checkpoint artifacts at configured intervals.
6. Evaluate checkpoints and write analysis bundles for later inspection.

The runtime also exposes live telemetry for the TUI, including:

- gate weights
- regime label
- expert routing history
- financial metrics
- gate accuracy window

Pause and resume are supported during interactive runs, and completed runs retain checkpoint-based analysis so the UI can still show regime and expert information after training ends.

## Artifact layout

Every run is written under `backend/artifacts/` in a directory named like `run-20260418T101220Z`.

The main files are:

- `summary.json`
  - Run-level metadata such as status, config, current episode, pause state, and runtime info.

- `metrics.json`
  - Episode-by-episode training series.

- `checkpoints/index.json`
  - Index of checkpoint summaries for the run.

Inside each checkpoint directory:

- `summary.json`
  - Checkpoint-level evaluation summary.

- `episode.json`
  - Full rollout record for the evaluation episode.

- `policy.json`
  - Policy outputs and action distribution data.

- `embedding.json`
  - Representation snapshot used for analysis.

- `regime_analysis.json`
  - Regime alignment statistics such as NMI, ARI, gate entropy, and specialization metrics.

- `expert_analysis.json`
  - Expert utilization and activation analysis.

The experiment runner also writes report bundles under `backend/artifacts/_experiments/<suite>/<timestamp>/`.
