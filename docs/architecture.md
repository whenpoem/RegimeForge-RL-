# RegimeForge Architecture

RegimeForge is the public name of the project. The installable Python package is `regime_lens`.

## What the system is

RegimeForge is a regime-aware reinforcement learning workbench for trading research.
It combines:

1. Configurable market environments with hidden regime dynamics.
2. Discrete and continuous trading agents, including gated mixture-of-experts models.
3. A training and evaluation pipeline that writes structured artifacts for analysis and resume.
4. TUI, Web, experiment runner, and report tooling built on the same artifact layout.

## Market environments

The discrete market is generated by `SyntheticMarketEnv` in `backend/regime_lens/market.py`.
Continuous single-asset and multi-asset markets live in `backend/regime_lens/continuous_market.py`.

At a high level, the environment works in two stages:

1. It samples a hidden regime sequence, normally using a Markov transition matrix.
2. It generates returns from regime-specific parameters such as drift, volatility, autocorrelation, and rare jumps.

The default regime set is:

- `bull`
- `bear`
- `chop`
- `shock`

Each regime has its own characteristics. For example, `bull` has positive drift, `bear` has negative drift, `chop` tends to be mean-reverting, and `shock` has higher volatility and jump risk.

The discrete environment exposes:

- A compact observation vector with engineered features such as short-term return, medium-term return, volatility, trend gap, drawdown, current position, and unrealized PnL.
- An oracle observation mode that appends the true regime as one-hot input.
- A regime schedule override for curriculum and OOD-style experiments.

The continuous environments expose allocation actions in `[-1, 1]`, multi-asset position vectors,
transaction/slippage costs, and optional non-stationary drift modes.

Real-data mode does not replay historical prices directly. It fits regime transition and
drift/volatility parameters from local price CSVs, then injects those parameters into the
synthetic environment for semi-realistic experiments.

## Agent variants

RegimeForge supports these discrete agent types, defined in `backend/regime_lens/config.py`:

- `dqn`
  - Standard DQN baseline.
  - No explicit regime input, gate, or experts.

- `rcmoe_dqn`
  - Regime-aware mixture-of-experts DQN.
  - Uses a gating network to route observations to multiple experts.
  - This is the main research model.

- `oracle_dqn`
  - DQN with the true regime one-hot appended to the observation.
  - Serves as an upper-bound reference.

- `hmm_dqn`
  - Two-stage baseline.
  - Uses a lightweight Gaussian mixture detector as the current HMM-style proxy baseline.
  - Feeds inferred regime probabilities to a DQN policy.

The discrete action space is:

- `short`
- `flat`
- `long`

Continuous experiments use `PPO`, `SAC`, and gated RCMoE actor-critic variants over allocation
weights. The same experiment runner can execute discrete or continuous smoke, benchmark, ablation,
and OOD suites.

## Training pipeline

Training lives in `backend/regime_lens/training.py` and is orchestrated by `TrainingManager`.

The pipeline is:

1. Create a new run directory and persist the initial summary.
2. Configure runtime settings such as device, CPU threads, and process priority.
3. Instantiate the selected environment and agent.
4. Roll out episodes, collect rewards, losses, and per-step trading statistics.
5. Flush metrics and checkpoint artifacts at configured intervals.
6. Evaluate checkpoints and write analysis bundles for later inspection.

The runtime also exposes live telemetry for the TUI, including:

- gate weights
- regime label
- expert routing history
- financial metrics
- gate accuracy window

Pause and resume are supported during interactive runs. Checkpoints persist model weights, replay
or rollout state, RNG state, config snapshots, stats, and reproducibility metadata so a run can be
continued or re-evaluated later.

## Artifact layout

Every run is written under `backend/artifacts/` in a directory named like `run-20260418T101220Z`.

The main files are:

- `summary.json`
  - Run-level metadata such as status, config, current episode, pause state, and runtime info.

- `metrics.json`
  - Episode-by-episode training series.

- `checkpoints/index.json`
  - Index of checkpoint summaries for the run.

Inside each checkpoint directory:

- `summary.json`
  - Checkpoint-level evaluation summary.

- `episode.json`
  - Full rollout record for the evaluation episode.

- `policy.json`
  - Policy outputs and action distribution data.

- `embedding.json`
  - Representation snapshot used for analysis.

- `regime_analysis.json`
  - Regime alignment statistics such as NMI, ARI, gate entropy, and specialization metrics.

- `expert_analysis.json`
  - Expert utilization and activation analysis.

- `stats.json`
  - Evaluation seeds and aggregate return baselines.

- `resume_state.json`
  - Episode, global step, checkpoint version, and saved config.

- `repro.json`
  - Runtime, Python/platform, checkpoint version, and tracking backend status.

- `explainability.json`
  - Policy/gate decision boundaries, attribution, counterfactuals, and transition-lag analysis when available.

- `weights/`
  - Model and optimizer state used for evaluation or resume.

The experiment runner also writes report bundles under `backend/artifacts/_experiments/<suite>/<timestamp>/`.
